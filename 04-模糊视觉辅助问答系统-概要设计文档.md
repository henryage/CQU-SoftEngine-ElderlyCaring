## 第一部分 引言

### 一、编写目的

编写本文的主要目的是把需求分析得到的用例模型转换为软件结构和数据结构。设计软件结构的具体任务是：将一个复杂系统按功能进行模块划分、建立模块的层次结构及调用关系、确定模块间的接口及人机界面等。数据结构设计包括数据特征的描述、确定数据的结构特性、以及数据库的设计。

本设计是指导详细设计和项目实施的重要指导性文件，也是后续详细设计、项目实施、系统集成测试的核心指导文件以及系统集成测试的重要依据。

### 二、读者对象

该文档的读者为用户代表、软件分析人员、开发管理人员和测试人员。

### 三、术语与缩写解释

1\.视觉问答（VisualQuestionAnswering，VQA）：人工智能多模态理解的重要任务，要求系统同时理解图像内容和自然语言问题，并给出符合场景的答案；

2\.图像识别：计算机通过算法对数字图像进行预处理、特征提取、特征匹配与推理分析，自动解析图像内容，识别出图像内物体类别、目标位置、文字、场景、物体属性等视觉信息的技术，本质是让机器模拟人类视觉感知能力看懂图像。

3\.多模态大模型：可统一处理文本、图像、音频、视频等多种媒介数据，完成跨模态理解、推理与生成的预训练大模型。

4\.长期记忆：独立于模型临时上下文的持久存储模块，用于保存跨会话历史交互、图像记录、用户偏好，通过检索复用历史信息，实现长期连贯智能交互。

## 第二部分 项目概述

### 一、项目描述

随着老龄化社会的加剧，老年群体在融入数字化生活时面临着难以跨越的"数字鸿沟"。老年人普遍存在视力衰退、手部震颤、认知负荷有限等生理特征变化，在阅读药品说明书、识别食品标签、理解医疗检查报告等日常健康场景中遇到诸多困难。同时，子女因工作繁忙难以随时陪伴，社区医疗资源紧张，老年人急需一款能够提供即时、便捷、低门槛健康辅助的智能工具。针对这一问题，本项目提出一种面向老年人关爱场景的模糊视觉辅助问答系统。系统以"棕色小猫"桌宠作为长期陪伴角色，以自然语言和语音为核心交互方式，降低老年人操作门槛。在日常交流过程中，系统通过多模态输入分析、图像识别与长期记忆机制，持续收集和更新老人的健康状况、用药习惯与生活轨迹信息，构建动态健康画像。系统内部采用角色包与知识包解耦架构设计，Agent运行时结合长期记忆与知识包能力，在对话过程中主动发现老人健康信息缺失项，并以自然交流的方式逐步获取相关信息。基于动态健康画像，系统能够针对老人当前状态提供个性化的用药提醒、健康建议与情感陪伴服务，同时支持子女或医生通过后台远程查看老人健康数据，构建"老人前端极简使用，子女/医生后台深度运维"的双角色闭环。

### 二、项目功能描述

用户业务主要包含智能视觉问答和本地记忆管理两个模块。

智能视觉问答模块除常规多模态问答功能外，还包含三个附加功能：提示词与接口管理、图像本地预处理。 本地记忆管理模块主要包含三个功能模块：历史问答记录模糊查询、记忆数据编辑与导出、跨会话重新识别。

#### 1\.智能视觉问答

该部分为本系统的核心功能模块。用户可以输入屏幕截图、本地图片、语音提问或文本等不同类型的数据，系统本地后端将对数据进行初步清洗与特征比对，调取本地关联知识，随后向外部大模型发起算力推理请求，将最终生成的智能应答流式反馈到前端桌宠用户界面中，得到最后的辅助问答结果，同时联动棕色小猫动作进行形态反馈。**针对老年人高频场景**，系统重点支持药品包装识别与用药指导、食品标签与保质期解读、日常物体辨认等辅助功能。

#### 2\. 提示词与接口管理

除常规多模态问答外，用户高级配置端可以在本地选择和切换不同的大模型 API 接口、基座大模型规模，或动态修改核心系统提示词。目的在于用户可以根据不同的任务场景，选择逻辑推理能力更强、响应速度更快、或者更贴合特定领域的接口，本地核心引擎将实时记录、存储并应用不同的接口配置参数。

#### 3\. 图像本地预处理

除常规多模态问答外，在数据离开用户单机设备前，系统利用本地 OpenCV 图像处理管道对用户截取的模糊或低质量图片进行实时处理。**针对老年人手抖、视力衰退导致的图像模糊问题**，系统自动触发的亮度增强、降噪和去模糊等参数调整，以得到最优质量的待识别图像，本地引擎将实时保存预处理后的图像特征并用于大模型的高效识别。

#### 4\.历史问答记录模糊查询

用户可以在本地查看自己的所有历史交互与问答记录，同时可以对给出的查询字段进行模糊查询，从而快速检索所需的历史知识。查询字段可以是：

1） 交互时间

2） 输入数据类型（截图/本地图片/语音/纯文本）

#### 5\. 记忆数据编辑与导出

记忆数据编辑与导出是指两个部分。其一（编辑），用户或管理员可以对本地 SQLite/向量数据库中的跨会话长期记忆、用户偏好和历史表单结果进行手动纠正或编辑，系统需要对编辑后的长期记忆进行本地覆盖，以修正后续的 RAG 知识检索偏好；其二（导出），用户可以将结构化的问答记录或运营审计日志表单打包导出，将其下载、迁移或备份到自己的其他本地终端设备上。

#### 6\.跨会话重新识别

重新识别是指对已经交互过的历史数据（历史截图、存量文本提问等）联合其当时的上下文进行再次识别与二次推理。识别过程与原始智能视觉问答过程大致相同，用户可以更换新的提示词或接口进行比对，系统需要将重新识别并经过本地安全洗净后的最终结果，覆盖原有的本地历史表单与记忆上下文。

## 第三部分 设计约束

### 一、需求约束

#### 1、本系统应当遵循的技术标准

数据编码与代码命名规则严格遵循《Google Python Style Guide》及《PEP 8 \-\- Style Guide for Python Code》中相关的规定。

#### 2、软、硬件环境标准

本系统采用完全单机落地的桌面 C/S（客户端/本地后台）架构。前端基于 PySide6/PyQt5 编写，本地核心引擎基于 Python 3\.9\+ 编写，本地持久化数据库采用 SQLite 及轻量化向量数据库。

系统部署在 Microsoft Windows 10/11（64位）或 macOS 12 版本以上的设备里。

#### 3、接口/协议标准

本系统在本地回环环境内通过进程间轻量级通信（RESTful API / WebSocket）实现桌宠前端和本地后台服务之间的数据通信；与外部公网通信时，通过标准 HTTPS 协议（TLS 1\.3 加密）及 RESTful 规范实现本地引擎对第三方大模型云平台 API 的出站调用。

#### 4、用户界面标准

使用 PySide6 组件库进行无边框、半透明窗体开发，融入桌面交互式动画（棕色小猫动态动作包），并进行界面全部美工视觉优化与气泡框高亮设计。**界面采用适老化设计规范**，包括18pt及以上大字体、高对比度配色、大按钮防误触及语音播报优先策略。

#### 5、软件质量

##### **1） 正确性** 

系统必须保证对用户输入的截图和文本能够被正确预处理，且本地合规拦截引擎能够 100% 正确触发涉及医疗、药品信息的安全提示。

##### **2） 健壮性** 

作为单机挂机常驻桌宠应用，系统应能够 7\*24 小时低功耗、无故障在系统后台或桌面运行，在断网或大模型 API 超时时能够自动捕捉异常并触发对应的桌宠动作反馈。

##### **3） 效率性** 

在单机环境中，OpenCV 图像管道对屏幕截图的预处理时间不超过 1 秒钟；在网络畅通情况下，流式首字响应（TTFT）时间不超过 2 秒钟。

##### **4） 易用性** 

界面应采用桌面宠物与图形化气泡框相结合的操作方式，支持一键截图与语音提问，以语音和自然语言为核心交互方式，降低老年人操作门槛。大字体、高对比度、语音播报与桌宠引导相结合，降低老年人认知负担，便于视障及老年用户无障碍使用。

##### **5） 安全性** 

用户的所有历史聊天记录、长期记忆、偏好及审计日志均以单机私有化密文或本地文件形式存储于本地设备；向外发起 API 调用时，核心报文与图片均采用 HTTPS 加密传输。对药品、医疗类内容自动追加风险提示，保障老人用药安全。

##### **6） 可扩展性** 

核心处理模块应采用解耦设计（如 Agent 编排模式），充分考虑到将来大模型接口的切换（如从 Open AI 格式切换为其他国产大模型 API）或提示词（Prompt）的升级，避免接口变更时大规模修改底层程序。

### 二、隐含约束

##### **1） 认知与操作技能** 

用户仅需具有最基本的电脑使用常识（如会用鼠标截图、能正常说话或打字）。对系统提供的常驻桌宠交互界面，应保证老年人无需经过复杂培训即可无障碍、直观地进行操作，以语音和一键操作为主，降低认知与操作门槛；

##### **2） 跨平台流畅度** 

系统必须针对 Windows 10/11 等系统进行内存轻量化优化，确保桌宠前端即使处于长时间挂机状态，其本地物理内存占用也不得对其他日常办公软件的流畅运行造成干扰；

##### **3） 配置灵活性** 

大模型 API 密钥（API Key）、接口终结点（Endpoint）、核心系统提示词（System Prompt），必须存放到本地的加密配置文件或本地数据库中，保证管理员或用户修改系统核心参数的灵活性。

## 第四部分 应用方案设计

### 一、安全设计

#### 1、用户隐私保护

系统在运行过程中会保存老人健康信息、用药记录、对话记录及长期记忆信息，为保护用户隐私，确立了以下原则：

1）老人健康画像数据仅用于个性化健康辅助服务

2）后台管理界面不能展示敏感原始数据

3）用户可对长期记忆内容进行查看、修改及删除

#### 2、数据存储安全

系统将用户信息、记忆数据及知识包数据分别存储，对于关键数据：

1）采用数据库统一管理

2）定期进行数据备份

3）对异常数据进行日志记录

4）防止数据误删或意外丢失

#### 3、输入安全控制

由于系统支持用户的多模态输入，因此需要对输入内容进行基础校验，如图片格式、文件大小、输入文本长度等，避免异常文件或恶意输入影响系统正常运行。同时在模型回答过程中，对不符合规范的输出进行过滤与修正，降低错误信息传播风险。对药品、医疗类内容自动追加安全风险提示。

#### 4、运行安全设计

系统采用模块化结构，各模块独立运行。

当图像分析、语言识别或知识检索等模块出现异常时：记录错误日志并返回提示信息，进而保证主程序继续运行。避免单一模块故障导致整个系统崩溃，提高系统稳定性和可靠性。

## 第六部分 总体设计

### 一、应用逻辑架构设计

#### 1、分层架构设计

系统采用"用户交互层—智能服务层—数据管理层"的三层应用逻辑架构，实现多模态交互、动态健康画像构建、视觉辅助问答以及长期陪伴服务等功能。

在用户端，老人可通过文本、语音和图像等方式与桌宠进行交互。系统接收用户输入后，结合图像内容分析、历史记忆以及用户健康画像信息生成个性化回复。

在智能服务层，Agent负责统一调度各业务模块，包括视觉辅助问答、健康画像分析、长期记忆管理、知识包调用和个性化健康辅助与陪伴等功能。系统通过长期对话持续收集老人健康特征，并动态更新健康画像权重，实现个性化健康辅助服务能力。

在数据管理层，系统负责用户数据、长期记忆、知识包数据及健康辅助记录等信息的存储与管理，为智能服务提供数据支撑。

### 二、技术架构设计

#### 1\.多模态交互模型设计

多模态交互模型负责统一处理用户输入信息，并将不同模态的数据转换为标准化推理请求。

##### 1）输入采集层

负责接收用户输入信息，包括：

- 文本输入 、语音输入 、图像输入、 用户操作事件

##### 2）预处理层

负责对原始输入数据进行标准化处理。

文本输入：

- 格式规范化 、 特殊字符过滤 

语音输入：

- 语音识别 、 文本转换 

图像输入：

- 图像缩放 、 图像归一化 、图像裁剪 、 图像增强 、噪声处理

**3）请求构建层**

负责将多模态信息统一组织为标准运行请求。

构建内容包括：

- 用户问题、 图像信息 、当前上下文 、用户状态信息

#### 2\.Agent运行模型设计

**1）上下文管理层**

负责整合推理所需上下文信息。

包括：

- 当前会话内容 、 长期记忆内容 、 用户画像信息 、知识包检索结果

**2）推理决策层**

负责生成推理计划并判断是否需要调用工具。

主要功能：

- 问题分析 、任务规划 、工具选择 、 工具调用控制

**3）回复生成层**

根据推理结果生成最终回答。

包括：

- 中性回答生成 、 角色风格转换 、 回复结构化输出

#### 3\.动态用户画像模型模型设计

动态用户画像模型用于持续构建和更新老人健康特征信息。

**1）特征采集层**

从用户交互过程中提取行为特征。

包括：

- 健康关注领域 、用药规律特征 、生活作息习惯 、情感状态变化

**2）权重计算层**

根据用户行为动态调整画像权重。

主要内容：

- 特征权重更新、特征衰减处理 、 特征冲突修正

**3）画像存储层**

负责用户画像数据持久化。

保存内容包括：

- 用户特征向量 、健康权重 、更新时间记录

#### 4\.长期记忆模型设计

长期记忆模型用于保存用户历史信息并辅助后续推理。

**1）记忆提取层**

从对话内容中提取重要信息。

包括：

- 用药记录 、健康事件 、生活偏好 、 特殊医嘱

**2）记忆管理层**

负责记忆更新与维护。

主要功能：

- 记忆新增 、记忆合并 、记忆删除 、记忆修正

**3）记忆检索层**

根据当前问题检索相关历史记忆，为Agent推理提供上下文支持。

#### 5\.知识包扩展模型设计

## 第七部分 用户界面设计

本系统的用户界面设计核心围绕"适老化"、"陪伴感"与"无障碍交互"展开。考虑到系统主要面向老年人群体，同时包含有视觉障碍或需要辅助交互的特殊群体，界面整体遵循高对比度、低认知负载以及强语音/动作反馈的设计原则。界面采用18pt及以上大字体、大按钮防误触、语音播报优先策略，最大程度降低老年人操作门槛。

### 一、界面布局设计

系统界面主要分为“前端桌宠主交互界面”与“本地后台管理控制面板”两大核心视图。

#### 前端桌宠主交互界面（常驻桌面端）

前端基于 PySide6/PyQt5 开发，采用无边框、半透明的悬浮窗体布局。

桌宠展示区：位于界面的核心视觉焦点，用于渲染"棕色小猫"的动态动作包（如倾听、思考、说话、报错等状态）。

交互气泡框：悬浮于桌宠上方或侧边。当大模型流式返回生成的原生答案时，文本将在这里以高亮、大字体（18pt及以上）的形式进行展示，辅以打字机动效。

极简输入区：隐藏式或半透明的输入悬浮条。提供醒目的大尺寸"一键截图"与"语音输入"按钮，支持全局快捷键唤醒及语音优先交互，方便老年及视障辅助人员进行"盲操"提问。

#### 历史记录与记忆管理面板

该面板采用传统的高效信息流布局，用于数据的检索与管理。

顶部检索区：提供时间维度筛选器（具体日期、时间段）和数据类型筛选器（截图、纯文本、语音等输入源分类）。支持智能关键字输入框。

主列表展示区：以数据表格或卡片列表的形式异步加载多模态记录（包括历史截图缩略图、提问文本及模型最终应答）。

状态标签列：每条记录附带彩色状态标签渲染（如"常规问答"、"医疗拦截"等），便于用户直观审阅及后续记忆编辑。

#### 高级配置与管理控制台

采用多标签页（Tab）侧边栏/顶部栏结构，整合系统的所有高级设置。根据后台设计，主要划分为以下几个选项卡：

模型与语音（TTS）：配置大模型 API 终结点、密钥以及本地/云端 TTS（文本转语音）引擎的参数（如参考语言、超时时间）。

权限与工具：控制模型主动交互的频率（如主动检查间隔、主动打招呼冷却）及浏览器插件（如 Playwright）的权限。

插件与系统：管理插件的启用状态与优先级，配置系统级输出（如输出终端运行日志、输出文件运行日志）。

记忆管理：展示当前正在读取或已存储的长期记忆 ID、内容摘要及更新时间。

### 二、业务界面风格展示

#### 视觉交互风格

轻量与沉浸感：主界面摒弃传统软件的厚重边框，全面采用异形透明窗口技术。桌宠如同真实存在于用户桌面上，不会干扰其他日常办公软件的正常使用界限。

适老化色彩设计：对于文本气泡框，采用高对比度的配色方案（如深色模式下的白字或亮黄字），字体大小不低于18pt。对于涉及医疗或风险的合规拦截提示，使用醒目的警示色（如红色或橙色）进行追加高亮渲染。

#### 动效与多模态反馈设计

状态联动引擎：界面的每一次状态变化均与桌宠动作强绑定。例如，触发一键截图并交由 OpenCV 预处理时，桌宠表现为"观察"或"快门"动作；大模型 API 测算推理时，桌宠进入"思考"循环动画；接收到文本流时，桌宠嘴部动作与 TTS 语音播报严格同步。

异常状态反馈：在断网或外部大模型 API 响应超时的情况下，系统自动捕捉异常，气泡框展示友好的错误提示，并触发桌宠特定的"困惑"或"安抚"动作，降低用户的操作挫败感。

#### 辅助交互设计

全键盘/快捷键驱动：为满足弱视或行动不便人群的需求，系统内所有核心操作（截图、录音、终止对话、调出记忆面板）均配有全局自定义快捷键。

焦点跟随与屏幕阅读：配置界面（尤其是管理控制台的各类表单）支持标准的 UI 焦点轮转，且相关按钮和状态标签均带有标准的无障碍语义标签，兼容主流操作系统的屏幕阅读器功能。

语音优先交互：针对老年人打字困难的情况，系统默认以语音输入为核心交互方式，TTS语音播报同步输出回答内容，实现"说话即提问，听到即回答"的无障碍体验。
